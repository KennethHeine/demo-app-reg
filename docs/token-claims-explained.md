# Token Claims Explained

This document explains the most important claims in the exported customer JWT files and how the backend uses them.

The example tokens live in `token-examples` and were generated by `scripts/export-jwt-examples.ps1`.

## Why These Claims Matter

The backend does not just check that a token exists. It validates a small set of claims that together answer four questions:

1. Was this token issued by the expected Entra tenant?
2. Was this token issued for this backend API?
3. Is the calling app allowed to call the API?
4. Which customer does that calling app belong to?

Those checks are implemented in `backend/app/auth.py`.

## Header Claims

### `alg`

- Example: `RS256`
- Meaning: the signing algorithm used for the JWT.
- Backend usage: the backend expects an RSA-signed token and verifies it with the tenant signing keys.

### `kid`

- Example: `QZgN9HqNkGNEM4GeKczD02PcVv4`
- Meaning: key identifier for the signing key.
- Backend usage: the backend uses this to select the correct key from the tenant JWKS document before verifying the signature.

### `x5t`

- Meaning: X.509 thumbprint of the signing certificate.
- Backend usage: not used directly by the backend, but useful when comparing signing metadata.

## Payload Claims The Backend Uses Directly

### `aud`

- Example: `api://kscloud.io/demo-app-reg-backend-api`
- Meaning: the intended audience of the token.
- Backend usage: the backend accepts only tokens issued for its configured API identifier URI or backend client id.
- Failure mode: if `aud` is wrong, token validation fails with a 401.

### `iss`

- Example: `https://sts.windows.net/14bc2ff7-5fd1-4ce2-a110-4f71b9a2ce41/`
- Meaning: the issuer that created the token.
- Backend usage: the backend allows tenant-specific Entra issuer forms for this demo.
- Important detail: these client-credentials tokens are currently arriving as v1 tokens in this tenant, so the issuer is the `sts.windows.net` form instead of only the v2 endpoint form.
- Failure mode: if `iss` is not in the allowed issuer set, the backend returns 401.

### `tid`

- Example: `14bc2ff7-5fd1-4ce2-a110-4f71b9a2ce41`
- Meaning: the Entra tenant id.
- Backend usage: the backend requires the token to come from the configured tenant.
- Failure mode: if `tid` is different, the backend returns 401.

### `roles`

- Example: `Customer.Data.Read`
- Meaning: application roles granted to the calling app for this API.
- Backend usage: the backend requires `Customer.Data.Read` before it will serve any customer data.
- Failure mode: if the role is missing, the backend returns 403.

### `appid` and `azp`

- Example: `3e82d5de-3cbe-44cf-afd4-242ff1d3fdb5`
- Meaning: the calling application id.
- Backend usage: the backend reads `azp` first and falls back to `appid`, then maps that id through `backend/customer-registry.local.json`.
- This is what determines which customer data is returned.
- Failure mode: if the app id is not found in the registry, the backend returns 403.

## Payload Claims That Are Important For Interpretation

### `exp`, `nbf`, and `iat`

- Meaning:
  `iat` is issued at.
  `nbf` is not before.
  `exp` is expiration time.
- Backend usage: these are part of standard JWT validation during decode.
- Why they matter in the exported files: they show exactly when the example token was valid.

### `appidacr`

- Meaning: how the confidential client authenticated.
- In this repo:
  `1` appears on the secret-based customers.
  `2` appears on the certificate-based customer.
- Why it matters: this is the easiest claim-level way to see the difference between the two auth modes in the exported examples.

### `oid`

- Meaning: object id of the calling service principal in the tenant.
- Backend usage: not used for routing or authorization in this demo.
- Why it is useful: it helps correlate the token to the exact enterprise application object in Entra.

### `sub`

- Meaning: subject identifier for the token.
- In these app-only tokens it matches the calling service principal identity rather than a human user.
- Backend usage: not used directly in this demo.

### `ver`

- Example: `1.0`
- Meaning: token version.
- Why it matters: it explains why the issuer is the v1-style Entra issuer in these client-credentials examples.

## Claim Differences Between Secret And Certificate Customers

The biggest visible difference in the example files is `appidacr`.

- `customer-python` and `customer-typescript` use client secrets and show `appidacr: "1"`.
- `customer-python-cert` uses a certificate and shows `appidacr: "2"`.

Everything else that matters to the backend is the same shape:

- same audience
- same tenant
- same required role
- different caller app id

That is exactly what you want: the backend should not need separate logic for secret-based versus certificate-based customers.

## How The Backend Uses The Claims In Order

The backend validation flow is effectively:

1. Read `kid` from the JWT header.
2. Resolve the correct signing key from the tenant JWKS.
3. Verify signature and audience.
4. Check `iss` against allowed issuer values.
5. Check `tid` against the configured tenant.
6. Check that `roles` contains `Customer.Data.Read`.
7. Read `azp` or `appid`.
8. Map that app id to a customer id through the generated registry.
9. Return only that customer’s records.

## How To Read The Exported Token Files

When you open an exported token example, focus on these fields first:

1. `aud`: proves the token was issued for the backend API.
2. `appid` or `azp`: identifies which customer app called the API.
3. `roles`: proves the app was granted backend access.
4. `tid`: proves the token belongs to the expected tenant.
5. `appidacr`: shows whether the client used secret or certificate auth.

## Common Troubleshooting Patterns

### Wrong `aud`

- Symptom: backend returns 401.
- Likely cause: the client requested the wrong scope.

### Missing `roles`

- Symptom: backend returns 403.
- Likely cause: the app registration has not been granted the backend application role.

### Unknown `appid`

- Symptom: backend returns 403 even though the role is present.
- Likely cause: the app exists in Entra but is missing from `customers.json`, so the generated customer registry does not contain it.

### Wrong `tid` or `iss`

- Symptom: backend returns 401.
- Likely cause: token came from the wrong tenant or an unexpected issuer form.
